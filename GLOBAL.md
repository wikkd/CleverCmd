# Minecraft Agent Mod 总体设计

## 1. 目标

本模组的目标，是让玩家用自然语言指挥一个受约束的 Agent，把语言意图转换成可审计、可确认、可撤销的游戏操作。

核心原则：

- 自然语言只负责表达意图，不直接等价于命令。
- 执行前必须经过结构化解析、权限校验和安全策略。
- 模型只是推理后端之一，不是唯一入口。
- 所有动作都必须落在预先注册的能力目录中。

## 2. 范围

### 2.1 覆盖内容

- 聊天框/面板输入自然语言。
- 意图识别、槽位抽取、歧义处理。
- 生成结构化执行计划。
- 命令执行与结果反馈。
- 确认、取消、撤销、超时处理。
- 审计日志与状态持久化。

### 2.2 不覆盖内容

- 不做无限自由对话型聊天机器人。
- 不允许模型直接越权执行未注册动作。
- 不把 LLM 输出当作最终执行指令。

## 3. 平台假设

- 以 Fabric 1.21.1 为初始落地目标。
- 设计保持平台中立，后续可迁移到其他模组/插件架构。
- 客户端负责输入、预览和确认。
- 服务端负责最终权限判断与真实执行。
- 支持中文优先，兼容英文。
- 支持“规则解析 + 模型推理”的混合方案。

## 4. 总体架构

### 4.1 流程

`输入层 -> 语义层 -> 规划层 -> 策略层 -> 编译层 -> 执行层 -> 反馈/状态层`

### 4.2 设计取舍

- 使用结构化中间表示，而不是直接输出可执行命令字符串。
- 计划和执行分离，先验证再执行。
- 统一动作目录，让解析、规划、执行共享同一份能力定义。

## 5. 核心模块

### 5.1 Input Gateway

负责接收玩家输入，入口可以是聊天框、快捷键或侧边面板。

### 5.2 Session Manager

负责一次自然语言回合的生命周期，包括等待确认、超时、取消和结束。

### 5.3 Context Collector

采集当前上下文，例如玩家位置、维度、朝向、背包、目标实体、目标方块、权限等。

### 5.4 NLU Engine

负责文本归一化、意图分类、槽位抽取、歧义检测。

### 5.5 Action Catalog

负责注册所有可执行动作、参数 schema、风险等级、权限需求。

### 5.6 Planner

负责把意图转换为可执行计划，必要时拆分成多步任务。

### 5.7 Policy Engine

负责权限、安全、速率、范围、确认规则判断。

### 5.8 Command Compiler

负责把抽象计划编译成命令字符串或原生 API 调用。

### 5.9 Execution Engine

负责任务调度、执行、重试、回滚和结果采集。

### 5.10 State Store

负责保存会话、偏好、任务、审计日志。

### 5.11 Feedback Renderer

负责向玩家输出解释、进度、失败原因和结果摘要。

### 5.12 Extension Manager

负责加载动作包、解析器、策略、后端适配器。

## 6. 自然语言解析与意图映射

### 6.1 流程

1. 文本归一化。
2. 意图族判定。
3. 槽位抽取。
4. 上下文补全。
5. 候选意图生成。
6. 评分与过滤。
7. 歧义处理。
8. 生成 `ActionPlan`。

### 6.2 规则与模型的关系

- 规则解析优先处理高频、明确、稳定的表达。
- 模型用于补足歧义、同义表达和复杂组合意图。
- 模型输出必须经过 schema 校验，不能直接执行。

### 6.3 示例

“把我传到最近的村庄”可拆为：

1. `locate village`
2. `verify target`
3. `tp self`

若权限不足或风险过高，先进入确认流程。

## 7. 指令执行链路

1. 预检上下文与动作 schema。
2. 策略判定权限与风险。
3. 必要时要求玩家确认。
4. 将计划编译为命令或原生操作。
5. 在正确线程执行。
6. 采集成功、失败、部分完成结果。
7. 回写状态与审计。
8. 不可逆失败则报告部分完成。

## 8. 权限与安全

### 8.1 权限模型

建议按能力范围而不是全局开关来设计：

- `self`
- `local_world`
- `target_entity`
- `admin`
- `world_edit`

### 8.2 安全原则

- 默认最小权限。
- 高危动作必须显式确认。
- 只允许白名单动作。
- 模型输出必须校验。
- 记录审计日志。

### 8.3 风险分级

- 只读
- 低风险
- 可逆
- 高风险
- 不可逆

## 9. 状态管理

建议至少分为五类状态：

- `SessionState`
- `ContextState`
- `PreferenceState`
- `ExecutionState`
- `AuditState`

状态键建议使用：

`playerId + worldId + dimension + sessionId`

原则：

- 服务端保存主状态。
- 客户端只保留缓存和显示层状态。
- 只保存结构化结果，不保存模型内部推理过程。

## 10. 错误处理

### 10.1 错误分类

| 错误类 | 触发点 | 处理策略 |
|---|---|---|
| 解析失败 | 句子太模糊 | 追问或给示例 |
| 意图冲突 | 多个动作候选 | 返回候选项 |
| 权限不足 | 需要更高权限 | 明确提示缺失权限 |
| 策略拒绝 | 高危或不可逆 | 要求确认或拒绝 |
| 执行失败 | 命令报错 | 报告具体步骤 |
| 后端不可用 | 模型/服务不可用 | 回退到规则解析 |
| 超时/冲突 | 任务过长或重叠 | 取消、排队或恢复 |
| 状态失配 | 世界状态变化 | 重新取上下文并重算 |

### 10.2 统一错误对象

建议包含：

- `code`
- `message`
- `severity`
- `recoverable`
- `suggested_action`
- `retry_after`

## 11. 扩展性设计

支持以下扩展点：

- 动作扩展
- 解析扩展
- 策略扩展
- 执行扩展
- 存储扩展
- UI 扩展

所有扩展都应包含：

- 版本号
- capability 声明
- 依赖声明
- 冲突检查

## 12. API 设计

以下 API 建议作为内部稳定契约，不直接耦合具体 Fabric 类。

### 12.1 AgentSessionApi

职责：

- 接收一次用户回合。
- 创建并返回 turn ID。
- 组织本回合生命周期。

输入：

- 玩家信息
- 原始文本
- 输入来源

输出：

- `turnId`
- 受理结果

调用时机：

- 玩家提交自然语言时。

依赖关系：

- `ContextApi`
- `StateApi`

### 12.2 ContextApi

职责：

- 采集当前游戏上下文。
- 生成结构化上下文快照。

输入：

- 玩家引用
- 世界引用
- 上下文标志位

输出：

- `ContextSnapshot`

调用时机：

- 解析前
- 执行前

依赖关系：

- 游戏运行时
- 权限提供者

### 12.3 ActionCatalogApi

职责：

- 注册和查询可执行动作。
- 提供参数 schema、风险等级和权限需求。

输入：

- 动作描述
- 查询条件

输出：

- 动作定义集合

调用时机：

- 启动时
- 扩展加载时

依赖关系：

- `ExtensionRegistryApi`

### 12.4 IntentParseApi

职责：

- 将自然语言映射为意图候选集合。

输入：

- 文本
- 语言
- 上下文
- 动作目录

输出：

- `IntentCandidateSet`

调用时机：

- 输入后
- 规划前

依赖关系：

- `ContextApi`
- `ActionCatalogApi`
- `ModelProviderApi`

### 12.5 ModelProviderApi

职责：

- 提供模型推理能力。
- 支持本地模型或远程模型适配器。

输入：

- 提示词
- schema
- 上下文片段

输出：

- 结构化预测结果

调用时机：

- 解析或规划需要推理时。

依赖关系：

- 模型后端适配器

### 12.6 PlanApi

职责：

- 将意图转换为可执行计划。

输入：

- 选中的意图
- 上下文
- 约束

输出：

- `ActionPlan`

调用时机：

- 歧义解决后。

依赖关系：

- `IntentParseApi`
- `ActionCatalogApi`

### 12.7 PolicyApi

职责：

- 执行权限和安全策略判定。

输入：

- `ActionPlan`
- 身份信息
- 策略配置

输出：

- `PolicyDecision`

调用时机：

- 执行前。

依赖关系：

- 权限系统
- `StateApi`

### 12.8 ConfirmationApi

职责：

- 请求并收集玩家确认。

输入：

- 策略结果
- 计划摘要

输出：

- `allow`
- `deny`
- `timeout`

调用时机：

- 高风险动作时。

依赖关系：

- `FeedbackApi`
- `StateApi`

### 12.9 CommandCompileApi

职责：

- 将 `ActionPlan` 编译为命令 bundle 或原生调用集合。

输入：

- `ActionPlan`
- 执行目标能力

输出：

- `CommandBundle`

调用时机：

- 确认通过后。

依赖关系：

- `ContextApi`
- `ActionCatalogApi`

### 12.10 ExecutionApi

职责：

- 调度并执行命令或原生操作。

输入：

- `CommandBundle`
- 取消令牌

输出：

- `ExecutionResult`

调用时机：

- 游戏线程或服务端执行时。

依赖关系：

- 命令分发器
- `StateApi`
- `FeedbackApi`

### 12.11 StateApi

职责：

- 读写会话、任务、偏好和审计状态。

输入：

- 查询键
- 状态补丁
- 快照

输出：

- 状态结果
- 写入结果

调用时机：

- 全流程。

依赖关系：

- 存储后端

### 12.12 FeedbackApi

职责：

- 输出解释、进度、错误和结果。

输入：

- 执行结果
- 错误对象
- 事件流

输出：

- 聊天消息
- 面板消息

调用时机：

- 每个阶段结束后。

依赖关系：

- 本地化系统
- UI 层

### 12.13 ExtensionRegistryApi

职责：

- 注册扩展并做兼容性检查。

输入：

- 扩展清单
- 版本信息
- capability 声明

输出：

- 注册结果

调用时机：

- 启动初始化时。

依赖关系：

- 所有注册表

## 13. 实现建议顺序

1. 先做 `ActionCatalog`。
2. 再做 `PolicyEngine`。
3. 再做 `ExecutionEngine`。
4. 然后补 `ContextCollector` 和 `StateStore`。
5. 最后接 `IntentParse` 和模型后端。

这样可以先保证“能安全执行”，再提升“能听懂人话”。

## 14. 结论

这个模组的本质不是聊天，而是一个受限的任务编排系统。  
真正的价值在于：让玩家用自然语言发起操作，同时保留结构化计划、安全控制和审计能力。
